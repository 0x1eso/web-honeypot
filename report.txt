1. 프로젝트 개요
------------------------------------------------------------------------------

1.1 프로젝트 소개

Web Honeypot은 일부러 취약해 보이는 가짜 웹 서버를 노출해 공격자를 유인하고,
유입되는 모든 HTTP 요청을 기록한 뒤 공격 유형을 자동 분류해 대시보드로
시각화하는 저상호작용 허니팟 시스템이다.

이 주제를 고른 동기는 정보보안에 대한 관심이다. 수업에서 SQL 인젝션, XSS
같은 공격 기법을 공격자 입장에서 배우는데, 그 공격이 방어자의 로그에는 실제로
어떤 모습으로 찍히는지 직접 데이터로 보고 싶었다. 허니팟은 공격 트래픽을
합법적으로, 그리고 안전하게 관찰할 수 있는 거의 유일한 방법이라 선택했다.

이 시스템은 실서비스가 아니라 학습용이며, 운영 범위를 격리 환경(로컬 Docker,
사설망/VLAN)으로 명시적으로 제한했다. 공인 IP 직접 노출, 운영망과 같은 서브넷
배치, 인증 없는 대시보드 공개는 금지 사항으로 README 최상단에 위협 모델로
명문화했다. 허니팟은 정의상 공격을 부르는 시스템이라 이 경계가 기능보다
먼저라고 판단했다.

1.2 담당 범위

팀 분담은 한 줄로: 팀원이 core(C 언어 수집기), 내가 api(Kotlin/Ktor)와
dashboard(React)를 맡았다. 내가 만든 범위는 (1) 수집된 로그를 5가지 공격
유형으로 분류하는 분류 엔진, (2) 로그/통계를 제공하는 REST API 4종,
(3) 이를 실시간으로 보여주는 통계 대시보드까지다. core 내부 구현(소켓,
파싱)은 팀원 몫이라 이 보고서에서는 데이터를 넣어주는 옆 모듈로만 다룬다.


2. 시스템 구성 / 아키텍처
------------------------------------------------------------------------------

2.1 전체 구조

시스템은 네 덩어리로 구성된다.

  - core   : C 수집기. 포트 8080을 공개해 공격을 받는 본체
  - SQLite : 공유 저장소. Docker named volume(/data/honeypot.db), WAL 모드
  - api    : Kotlin/Ktor 서버. 포트 8081, 내부 네트워크 전용
  - dashboard : React + nginx. 포트 3000, /api 요청을 api 컨테이너로 프록시

데이터 흐름은 한 문장으로: 공격자의 요청을 C가 받아 DB에 저장하고, Kotlin이
주기적으로 읽어 분류하고, React가 10초마다 폴링해 시각화한다.

2.2 핵심 설계 결정: 2-phase write

C 수집기는 요청 원문(IP, 메서드, 경로, User-Agent, 바디, 시각)을 attack_type
컬럼만 NULL로 비운 채 INSERT만 한다. 분류는 하지 않는다. Kotlin 분류기가
5초 주기로 미분류 행(attack_type IS NULL)을 읽어 빈칸만 UPDATE로 채운다.

이렇게 가른 이유는 두 가지다. 첫째, 허니팟은 공격자에게 빨리, 진짜 서버처럼
응답해야 의심을 사지 않는데 분류 연산이 응답 경로에 끼면 지연이 생긴다.
둘째, C와 Kotlin이 서로의 존재를 몰라도 되게 하려면 결합점을 DB 스키마
하나로 줄이는 것이 협업상 가장 단순했다.


3. 제작 절차 (시간순)
------------------------------------------------------------------------------

3.1 Phase 1 — Kotlin API 서버 구축

한 일: DB 연결(Database.kt), 분류 엔진(AttackClassifier.kt), 라우팅
(Routing.kt), DTO(Models.kt)로 파일을 역할별로 분리해 API 서버의 뼈대를
세웠다. 엔드포인트는 /api/logs(필터+페이지네이션), /api/stats(유형별 집계),
/api/top-ips(상위 공격 IP), /healthz(DB ping) 4개로 잡았다.

막힌 점 ①: Exposed ORM 0.52 버전에서 예전 자료에 나오는 문법이 통째로
컴파일되지 않았다. Table.select { 조건 } 도, .limit(n).offset(m) 체이닝도
"unresolved reference" 에러만 떴다. 처음에는 내 import 문제인 줄 알고
한참 헤맸다.

해결: 에러 메시지로는 원인을 못 찾아서 Exposed 공식 GitHub의 릴리스 노트와
breaking changes 문서를 확인했고, 0.52에서 구 API가 제거되고
selectAll().where { 조건 } 과 limit(n, offset) 형태로 바뀐 것을 알았다.
전체 쿼리를 새 문법으로 교체했고, 같은 함정을 팀이 다시 밟지 않도록
IMPLEMENTATION.md에 "0.52 API 주의사항" 섹션으로 구문법/신문법 대비표를
남겼다.

3.2 Phase 2 — React 대시보드 구축

한 일: StatCard(유형별 카운트), StatsChart(recharts 파이차트), TopIps(가로
막대), LogTable(필터+페이지네이션 표) 4개 컴포넌트를 만들고, 10초 간격
자동 갱신(setInterval + Promise.all 병렬 fetch)을 붙였다.

막힌 점 ②: 로컬 개발에서는 멀쩡하던 화면이 Docker로 배포하니 데이터가 전부
사라졌다. 개발자 도구를 열어 보니 모든 요청이 /api/api/logs 로 나가서 404를
받고 있었다.

원인: 환경변수로 주입되는 BASE 주소에 이미 /api가 들어 있는데, 호출부에서
또 `${BASE}/api/logs` 로 /api를 한 번 더 붙이고 있었다. 로컬 BASE에는 /api가
없어서 증상이 숨어 있다가 배포 환경에서만 터진 것이다.

해결: "BASE가 /api를 소유한다"로 정책을 통일하고 호출부는 `${BASE}/logs`
형태로 고쳤다. 그리고 다시는 안 생기게 Playwright e2e 테스트에서 페이지가
보내는 모든 요청 URL을 가로채 /api/api/ 가 포함된 요청이 0건임을 단언하는
회귀 가드를 박아뒀다.

3.3 Phase 3~4 — 성능·동시성·보안 보강

동시성: C의 INSERT와 Kotlin의 UPDATE가 같은 SQLite 파일에 동시에 들어가면
SQLITE_BUSY 충돌이 난다. WAL 저널 모드로 읽기/쓰기를 분리하고
busy_timeout=5000ms로 잠금 경합을 대기로 흡수했다. 분류도 한 사이클에
최대 500행(BATCH_LIMIT)만 처리해 긴 트랜잭션이 C의 쓰기를 굶기지 않게 했다.

성능: 인덱스 3개(attack_type / ip+timestamp 복합 / timestamp)를 추가했다.
특히 브루트포스 판정은 원래 로그인성 경로 패턴별로 쿼리를 4번 돌려 합산했는데,
같은 행이 두 패턴에 걸리면 중복 카운트되는 버그 가능성이 있어 OR로 묶은 단일
COUNT 쿼리로 바꿨고, (ip, timestamp) 인덱스 덕에 풀스캔도 사라졌다.

보안: CORS는 anyHost()를 금지하고 ALLOWED_ORIGINS 환경변수 화이트리스트로만
열도록 했다. nginx에는 CSP 포함 보안 헤더 5종을 적용했고, 세 컨테이너 모두
non-root 사용자로 바꿨다. 허니팟 응답은 전부 정적 문자열이라 입력이 응답에
반사되지 않는다(반사형 XSS 표면 0).

막힌 점 ③: non-root nginx(nginx-unprivileged) 이미지로 바꾸자 컨테이너가
"[emerg] open() /etc/nginx/conf.d/default.conf failed (13: Permission denied)"
를 뱉으며 부팅 루프에 빠졌다.

원인: Dockerfile의 COPY는 기본적으로 root 소유로 파일을 넣는데, 이 이미지는
nginx 프로세스가 uid 101로 돌아서 root 소유 설정 파일을 읽지 못했다.

해결: COPY --chown=nginx:nginx 로 설정 파일과 정적 산출물의 소유권을 명시해
해결했다. non-root 이미지에서 가장 흔한 첫 부팅 실패 원인이라는 것을 직접
밟아보고 알았다.

3.4 Phase 5 — 테스트 작성

Ktor의 testApplication 기반 통합 테스트 7개(엔드포인트 응답 형식, type 필터,
페이지네이션, 음수 offset 클램프, stats의 미분류 제외, top-ips 정렬, healthz),
분류기 단위 테스트 5개(SQLi/XSS/스캔/브루트포스/기타 각 패턴), Playwright
e2e 2개(이중 prefix 회귀 가드, 렌더 무에러)를 작성했다. 마지막으로 Docker
환경을 실제로 띄워 가짜 공격 페이로드를 보내고 분류→API→대시보드까지
이어지는 통합 검증 8항목을 전부 통과시켰다.


4. 결과물 설명
------------------------------------------------------------------------------

4.1 공격 분류 동작

다섯 유형을 각각 한 줄로 요약하면:

  - SQLi      : select, union, sleep(, 1=1 등 20개 키워드가 경로/바디에 포함
  - XSS       : <script, onerror=, javascript: 등 14개 패턴 포함
  - 스캔      : /.env, /wp-admin, /etc/passwd 등 민감 경로 19종으로 시작
  - 브루트포스 : 같은 IP가 60초 안에 로그인성 경로를 10회 이상 요청
  - 기타      : 위 어디에도 해당하지 않는 요청

매칭에는 우선순위가 있어(SQLi → XSS → 스캔 → 브루트포스 → 기타) 먼저 걸리는
유형으로 확정된다. 패턴은 일부러 느슨하게 잡았다. 허니팟에서는 공격을 놓치는
비용이 오탐 비용보다 크다고 판단했기 때문이다.

4.2 대시보드 화면

상단에 유형별 요약 카드 5장, 그 아래 유형 분포 파이차트와 상위 공격 IP 가로
막대 차트, 하단에 유형 필터와 페이지네이션(20행 단위)이 있는 로그 표가 있다.
전체 화면이 10초마다 자동 갱신되어 공격이 들어오는 즉시 수치가 움직인다.


5. 테스트 계획
------------------------------------------------------------------------------

검증 대상은 네 가지였다: API 응답 형식(필드/상태코드), 분류 정확도(유형별
대표 페이로드), 페이지네이션 경계값(음수 offset 포함), 화면 렌더(콘솔 에러 0).

자동화 구성은 Ktor 통합 7 + 분류기 단위 5 + Playwright e2e 2 = 14개이고,
별도로 실제 Docker 스택에 페이로드를 흘려보내는 통합 검증 8항목을 수행했다.

회귀 가드를 일부러 만든 이유: /api/api/ 404는 "로컬에서는 재현이 안 되는"
유형의 버그라 사람 눈의 리뷰로는 재발을 못 막는다. 그래서 페이지의 모든
네트워크 요청을 캡처해 이중 prefix가 0건임을 기계가 단언하게 했다. 테스트는
커버리지 숫자보다 한 번 내린 결정이 다시 무너지지 않게 잠그는 장치라는 걸
이 버그에서 배웠다.


6. 회고 / 배운 점
------------------------------------------------------------------------------

제일 어려웠던 점: 동시성이었다. 두 개의 다른 언어로 짠 프로세스가 하나의
SQLite 파일을 같이 쓰는 상황은 코드만 봐서는 문제가 안 보이고, 부하를 걸어야
SQLITE_BUSY가 터진다. WAL, busy_timeout, 배치 제한이 각각 무엇을 해결하는지
구분해서 이해하는 데 시간이 가장 많이 들었다.

협업에서 배운 것: C 팀원과 코딩 시작 전에 DB 스키마(컬럼/CHECK 제약),
포트(8080/8081/3000), 타임스탬프 형식(ISO 8601 UTC)을 먼저 합의해 둔 것이
결정적이었다. 결합점을 DB 하나로 줄여 둔 덕에 서로의 코드를 한 줄도 읽지
않고도 통합이 한 번에 됐다. 인터페이스 합의가 코드보다 먼저라는 걸 체감했다.

보안 관점에서 새로 알게 된 것: "공격을 부르는 시스템일수록 자기 자신은 더
단단해야 한다"는 역설이다. 위협 모델을 문서 최상단에 박는 것, non-root
컨테이너·CORS 화이트리스트·정적 응답 같은 다층 방어를 기본값으로 시작하는
것이 기능 추가보다 먼저라는 사고방식을 얻었다.

더 했으면 좋았을 것: 분류 규칙이 substring 매칭이라 정상 요청도 오탐될 수
있다. 정규식·스코어링 기반으로 고도화하고 싶다. 또 recharts 때문에 대시보드
번들이 596KB로 커서 코드 분할이 필요하고, 수집된 실데이터가 쌓이면 유형별
시계열 추이 같은 분석 화면도 붙여 보고 싶다.
