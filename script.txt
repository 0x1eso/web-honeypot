■ 슬라이드 1 — 타이틀  (약 60초)
------------------------------------------------------------------------------
안녕하세요, Team Zeraxis의 도현서입니다.

저희 팀이 만든 프로젝트는 Web Honeypot, 우리말로 하면 "웹 꿀단지"입니다.
이름 그대로입니다. 일부러 달콤해 보이는 — 그러니까 일부러 취약해 보이는 —
가짜 웹 서버를 세워두고, 거기에 꼬여드는 공격을 전부 기록하고, 자동으로
분류해서, 대시보드로 보여주는 시스템입니다.

(하단 다섯 개 색 점을 가리키며)
아래 보이는 다섯 가지 색이 오늘 발표 내내 반복해서 나올 다섯 가지 공격
유형입니다. SQL 인젝션, XSS, 스캔, 브루트포스, 그리고 기타. 색만 기억해
두셔도 뒤의 화면들이 편하게 읽히실 겁니다.

전체 시스템은 C, Kotlin, React 세 언어로 나뉘어 있고, 저는 그중에서
Kotlin API 서버와 React 대시보드를 맡았습니다. C 수집기는 팀원이
담당했습니다.

■ 슬라이드 2 — 문제 정의  (약 70초)
------------------------------------------------------------------------------
먼저 "왜 허니팟인가"부터 말씀드리겠습니다.

저희가 보안 수업에서 SQL 인젝션이나 XSS 같은 공격 기법을 배울 때는 항상
공격자 입장이었습니다. 그런데 문득 궁금해졌습니다. 이 공격이 방어자의
로그에는 실제로 어떤 모습으로 찍힐까? 진짜 인터넷의 봇들은 어떤 경로를,
어떤 페이로드로 두드릴까?

이걸 합법적으로, 안전하게 관찰할 수 있는 방법이 사실상 허니팟 하나입니다.

(카드 세 장을 왼쪽부터 차례로 가리키며)
동작은 세 단계입니다. 첫째 "유인" — 점-env 파일이나 wp-admin처럼 봇들이
좋아하는 미끼 경로를 노출하고, 가짜 200 응답을 줘서 공격자를 머물게 합니다.
둘째 "수집" — 들어오는 모든 요청의 IP, 메서드, 경로, User-Agent, 바디,
시각을 기록합니다. 셋째 "분석" — 이걸 다섯 유형으로 자동 분류해서
대시보드로 보여줍니다.

한 가지 분명히 해 둘 점은, 이건 실서비스가 아니라 학습용이고, 운영 범위를
격리망으로 제한해 설계했다는 겁니다. 이 얘기는 보안 슬라이드에서 다시
드리겠습니다.

[클릭]


■ 슬라이드 3 — 시스템 아키텍처  (약 90초)
------------------------------------------------------------------------------
전체 구조입니다. 왼쪽에서 오른쪽으로, 공격이 데이터가 되는 한 줄 흐름입니다.

(흐름도를 왼쪽부터 따라가며)
공격자의 HTTP 요청은 맨 먼저 C로 짠 core 모듈이 받습니다. 8080 포트로
공개돼 있는, 시스템에서 유일하게 바깥에 노출된 부분입니다. 이 부분은
팀원이 담당했고, 오늘 발표에서는 "데이터를 넣어주는 옆 모듈" 정도로만
다루겠습니다.

core는 받은 요청을 SQLite에 저장합니다. 여기서 중요한 점 하나 —
(하단 왼쪽 문구를 가리키며) core는 attack_type 컬럼을 비워둔 채, 원문
그대로 적재만 합니다. 분류는 하지 않습니다.

그 빈칸을 채우는 게 제가 만든 Kotlin API 서버입니다. 5초 주기로 미분류
행을 읽어서 분류하고 UPDATE합니다. 그리고 똑같이 제가 만든 React
대시보드가 10초마다 API를 폴링해서 화면을 갱신합니다.

(인디고 색 박스 두 개를 가리키며)
색이 칠해진 오른쪽 두 박스가 오늘 발표의 범위입니다.

참고로 api 서버는 8081 포트인데 호스트에 공개되지 않습니다. 대시보드의
nginx가 슬래시-api 요청을 내부 네트워크로 프록시하는 구조라, 외부에서
직접 닿을 수 있는 건 끝까지 core 하나뿐입니다.

[클릭]


■ 슬라이드 4 — 협업 인터페이스  (약 80초)
------------------------------------------------------------------------------
이번 장은 협업 이야기입니다. C를 쓰는 팀원과 Kotlin을 쓰는 제가 어떻게
충돌 없이 따로 개발했는가 — 답은 "모듈 간 계약을 DB 하나로 끝낸다"였습니다.

(왼쪽 카드를 가리키며)
방금 보신 구조를 저희는 2-phase write라고 부릅니다. 1단계, core가 원문을
INSERT하면서 attack_type은 NULL로 둡니다. 2단계, Kotlin이 그 빈칸만
UPDATE로 채웁니다. 이렇게 가르면 C와 Kotlin이 서로의 코드를 한 줄도 몰라도
됩니다. 결합점이 DB 스키마 하나뿐이니까요.

왜 이렇게 갈랐냐면 — 허니팟은 공격자에게 빨리, 진짜 서버처럼 응답해야
의심을 안 삽니다. 분류처럼 무거운 연산이 응답 경로에 끼면 안 되니까,
응답은 C가, 분석은 Kotlin이 나눠 맡는 게 자연스러웠습니다.

(오른쪽 카드를 가리키며)
대신 코딩 시작 전에 세 가지를 먼저 합의했습니다. DB 파일 경로, timestamp는
ISO 8601 UTC 포맷, 그리고 컬럼 길이 CHECK 제약은 스키마 파일·C 코드·Kotlin
코드 세 곳이 항상 같은 값을 갖는다 — 이 세 줄 약속 덕분에 마지막 통합이
한 번에 됐습니다.

아래 포트 규약도 같은 맥락입니다. 8080만 공개, 8081은 내부 전용, 3000이
대시보드 입구입니다.

[클릭]


■ 슬라이드 5 — REST 엔드포인트  (약 70초)
------------------------------------------------------------------------------
여기서부터 제 담당인 API 서버 내부로 들어갑니다.

엔드포인트는 네 개뿐입니다. 작지만 전부 방어적으로 만들었습니다.

(표를 위에서부터 한 줄씩 가리키며)
첫째, 슬래시 api 슬래시 logs. 로그 목록입니다. limit, offset, 유형 필터를
받아서 최신순으로 페이지를 돌려줍니다.
둘째, stats. 유형별 집계입니다. 아직 분류 안 된 행은 집계에서 빼고, 전체
건수에만 포함시킵니다.
셋째, top-ips. 공격 횟수 상위 IP 목록이고요.
넷째, healthz. 단순히 살아있다가 아니라, DB에 SELECT 1을 실제로 날려서
통과해야만 200을 줍니다. DB가 잠겼거나 파일 권한이 깨진 사고를 헬스체크
단계에서 잡기 위해서입니다.

(하단 각주를 가리키며)
방어적이라고 한 이유가 아래에 있습니다. limit은 1에서 500 사이로 강제
클램프, 음수 offset은 0으로 보정해서 어떤 이상한 값이 와도 500 에러가
나지 않습니다. 그리고 모든 핸들러가 newSuspendedTransaction으로 DB 작업을
IO 스레드풀에 격리합니다 — 이 이유는 7번 슬라이드에서 자세히 말씀드립니다.

[클릭]


■ 슬라이드 6 — 공격 분류 엔진  (약 90초)
------------------------------------------------------------------------------
이 프로젝트의 심장인 분류 엔진입니다.

(상단 칩 행렬을 왼쪽부터 가리키며)
분류는 우선순위 매칭입니다. SQLi부터 검사해서, 걸리면 거기서 확정.
안 걸리면 XSS, 스캔, 브루트포스 순서로 내려가고, 끝까지 안 걸리면
기타입니다. 한 요청에 SQL 키워드와 스크립트 태그가 둘 다 있어도 레이블은
SQLi 하나로 떨어진다는 뜻입니다.

(표를 위에서부터 짚으며)
규칙을 보시면 — SQLi는 union, sleep 괄호, 1=1 같은 키워드 20개를 경로와
바디에서 찾습니다. XSS는 script 태그, onerror 같은 패턴 14개. 스캔은
점-env, wp-admin, etc-passwd 같은 민감 경로 19종으로 시작하는지를 봅니다.

브루트포스만 성격이 다릅니다. 패턴이 아니라 빈도입니다. 같은 IP가 60초
안에 로그인성 경로를 10번 이상 두드리면 브루트포스로 판정합니다.

(하단 각주를 가리키며)
이 분류기는 5초 주기 코루틴이 돌리고, 한 사이클에 최대 500행까지만
처리합니다. 그리고 의도적으로 패턴을 느슨하게 잡았습니다. 정상 요청을
공격으로 잘못 분류하는 오탐이 있을 수 있는데, 허니팟에서는 공격을 놓치는
비용이 오탐 비용보다 훨씬 크다고 판단했기 때문입니다. 어차피 여기 들어오는
트래픽은 대부분 공격이니까요.

[클릭]


■ 슬라이드 7 — 동시성·안정성  (약 90초)
------------------------------------------------------------------------------
개인적으로 이번 프로젝트에서 제일 어려웠던 부분입니다.

문제의 본질은 제목 그대로입니다 — 하나의 SQLite 파일을 두 프로세스가
동시에 씁니다. C가 INSERT하는 동안 Kotlin이 UPDATE를 하면 SQLITE_BUSY
충돌이 납니다. 코드만 봐서는 안 보이고, 트래픽을 걸어야 터지는 종류의
문제라서 원인 파악에 시간이 가장 많이 들었습니다.

(카드를 시계 방향으로 가리키며)
해결은 네 겹입니다.
첫째, WAL 모드와 busy_timeout 5초. WAL은 읽기와 쓰기를 분리해 주고,
그래도 잠금이 겹치면 5초까지 기다렸다가 재시도합니다.
둘째, 아까 나온 BATCH_LIMIT 500. 분류 트랜잭션이 길어지면 그동안 C의
INSERT가 굶기 때문에, 일부러 500행에서 끊습니다.
셋째, 브루트포스 카운트 쿼리 최적화. 원래 경로 패턴별로 쿼리를 네 번
돌렸는데, 한 행이 두 패턴에 걸리면 중복 카운트되는 위험이 있었습니다.
OR로 묶은 단일 COUNT로 바꾸고 IP-타임스탬프 복합 인덱스를 깔아서 정확성과
속도를 같이 잡았습니다.
넷째, 종료 처리입니다. 분류 루프가 SupervisorJob 위에서 돌고, 서버 종료
신호를 구독하고 있다가 코루틴을 안전하게 취소합니다. 도커 재시작 때 분류가
중간에 끊겨도 데이터가 깨지지 않습니다.

(하단 각주)
그리고 모든 DB 접근이 Dispatchers.IO로 격리되는 이유 — Ktor는 적은 수의
이벤트루프 스레드로 모든 요청을 처리하는데, JDBC는 블로킹입니다. 격리하지
않으면 DB가 느려지는 순간 서버 전체가 멈춥니다.

[클릭]


■ 슬라이드 8 — 대시보드  (약 70초)
------------------------------------------------------------------------------
이제 수집·분류된 데이터를 보는 화면, React 대시보드입니다.

(왼쪽 목업을 가리키며)
왼쪽이 실제 화면의 축소판입니다. 구성은 네 가지 — 위에 유형별 카운트
카드 다섯 장, 가운데에 유형 비율 파이차트와 상위 공격 IP 막대 차트,
아래에 로그 표가 있습니다.

(오른쪽 카드를 위에서부터)
StatCard는 유형별 집계 카드인데, 색을 전부 디자인 토큰으로 관리해서
지금 보시는 발표자료의 색과 대시보드 색이 같은 값입니다. 차트 두 개는
recharts로 그렸고, LogTable은 유형 필터와 20행 단위 페이지네이션을
지원합니다.

(하단 각주)
갱신은 10초 주기 자동 폴링입니다. stats, top-ips, logs 세 요청이 서로
독립적이라 Promise.all로 병렬로 쏩니다. 순차로 기다리면 한 화면 갱신에
세 요청 시간이 다 더해지니까요. 공격이 들어오면 10초 안에 화면 숫자가
움직이는 걸 볼 수 있습니다.

■ 슬라이드 9 — 보안 설계  (약 80초)
------------------------------------------------------------------------------
앞에서 미뤄뒀던 보안 이야기입니다. 저희가 이 프로젝트에서 얻은 가장 큰
교훈을 한 문장으로 적었습니다 — 공격을 부르는 시스템일수록, 안쪽은 더
단단해야 한다.

(상단 박스를 가리키며)
먼저 위협 모델입니다. 이 시스템은 격리된 사설망과 로컬 도커 전용입니다.
공인 IP에 직접 노출, 운영망과 같은 서브넷 배치, 인증 없는 대시보드 공개 —
전부 금지이고, README 맨 위에 박아뒀습니다. 허니팟은 정의상 공격을
유도하는 물건이라, 어디까지 노출해도 되는가를 코드보다 먼저 정해야 한다고
생각했습니다.

(아래 카드 네 장을 차례로)
그 위에 네 겹의 방어를 쌓았습니다.
컨테이너 세 개 전부 non-root로 돌립니다. 뚫려도 root가 없습니다.
nginx에는 CSP를 포함한 보안 헤더 다섯 종을 적용했습니다.
CORS는 아무 origin이나 허용하는 anyHost를 금지하고, 환경변수 화이트리스트로만
엽니다.
마지막으로 허니팟의 응답은 전부 정적 문자열입니다. 공격자가 보낸 입력이
응답에 한 글자도 반사되지 않기 때문에, 반사형 XSS 표면이 구조적으로
0입니다.

[클릭]


■ 슬라이드 10 — 테스트  (약 70초)
------------------------------------------------------------------------------
테스트입니다. 저희가 강조하고 싶은 건 개수보다 부제 쪽입니다 — 테스트는
한 번 내린 결정이 다시 무너지지 않게 잠그는 장치라는 것.

(큰 숫자를 차례로)
구성은 Ktor 통합 테스트 7개, 분류기 단위 테스트 5개, Playwright e2e 2개.
그리고 오른쪽, 실제 도커 스택을 띄워 가짜 공격을 흘려보내고 분류부터
대시보드까지 검증하는 통합 체크 8항목 전부 통과입니다.

(하단 표를 가리키며)
아래 세 줄이 "잠근 결정"들입니다.
첫째 줄이 대표적인데요 — 뒤에서 말씀드릴 이중 prefix 404 버그를 잡은 뒤,
페이지가 보내는 모든 요청 URL을 캡처해서 api가 두 번 붙은 주소가 0건임을
기계가 단언하게 했습니다. 사람 리뷰로는 이런 회귀를 못 막으니까요.
둘째, 음수 offset 같은 경계값이 와도 500이 아니라 200이 나오는 것.
셋째, 집계에서 미분류를 빼기로 한 정책 자체를 테스트로 고정한 것.

[클릭]


■ 슬라이드 11 — 트러블슈팅  (약 100초, 발표의 하이라이트)
------------------------------------------------------------------------------
오늘 발표에서 제일 들려드리고 싶은 장입니다. 매끈하게 된 것보다, 막혔던
세 지점이요.

(왼쪽 카드)
첫 번째, Exposed ORM 0.52 버전. 인터넷 자료에 나오는 옛날 쿼리 문법이
통째로 컴파일이 안 됐습니다. 처음엔 제 import 문제인 줄 알고 한참
헤맸는데, 에러 메시지로는 답이 안 나와서 공식 릴리스 노트를 뒤졌더니
0.52에서 구 API가 아예 제거된 거였습니다. 신 문법으로 전부 교체하고,
팀이 같은 함정을 다시 밟지 않게 구문법-신문법 대비표를 문서로 남겼습니다.

(가운데 카드)
두 번째가 제일 기억에 남습니다. 로컬에서는 멀쩡하던 대시보드가 도커로
배포하니까 데이터가 전부 사라졌습니다. 개발자 도구를 열어 보니 모든
요청이 슬래시 api 슬래시 api 슬래시 logs — api가 두 번 붙어서 404를 받고
있었습니다. 원인은 단순했습니다. 베이스 주소에 이미 api가 들어 있는데
호출부에서 또 붙인 거죠. 로컬 환경의 베이스에는 api가 없어서 증상이
숨어 있다가 배포에서만 터진 겁니다. "베이스가 api를 소유한다"로 정책을
통일해서 고쳤고, 방금 보신 e2e 회귀 가드를 박아서 재발을 차단했습니다.

(오른쪽 카드)
세 번째, 보안 강화를 하다가 생긴 사고입니다. nginx를 non-root 이미지로
바꾸자마자 컨테이너가 permission denied를 뱉으며 부팅 루프에 빠졌습니다.
도커의 COPY가 기본적으로 root 소유로 파일을 넣는데, 이 이미지는 nginx가
일반 유저로 돌아서 자기 설정 파일을 못 읽는 거였습니다. COPY에 chown
옵션으로 소유권을 명시해서 해결했습니다. 보안을 올리면 운영 가정이 같이
바뀐다는 걸 몸으로 배웠습니다.

 부록 — 예상 질문 & 답변 메모
------------------------------------------------------------------------------
Q1. 왜 SQLite인가? PostgreSQL 같은 걸 안 쓰고?
A. 단일 파일이라 C와 Kotlin이 별도 DB 서버 없이 볼륨 하나로 공유할 수
   있고, 학습용 트래픽 규모에서는 WAL 모드로 충분합니다. 운영 규모가
   커지면 서버형 DB로 옮기는 게 맞습니다.

Q2. 분류를 머신러닝으로 안 한 이유는?
A. 규칙 기반이 설명 가능하고, 학습 데이터가 없는 초기 단계에 적합합니다.
   수집이 쌓이면 그 데이터가 곧 ML의 학습셋이 될 수 있습니다.

Q3. 공격자가 허니팟인 걸 눈치채면?
A. 가짜 200 응답, 위장된 Server 헤더 등으로 체류 시간을 늘리지만, 정교한
   공격자는 결국 알아챕니다. 저희 목표는 자동화된 봇 트래픽 수집이라
   그 수준의 위장으로 충분하다고 판단했습니다.

Q4. 브루트포스 기준(60초/10회)의 근거는?
A. 사람의 수동 로그인 실수로는 도달하기 어렵고 자동화 도구는 쉽게 넘는
   수준으로 잡은 운영 파라미터입니다. 상수로 분리돼 있어 조정 가능합니다.

Q5. core가 뚫리면?
A. core는 non-root 컨테이너이고 쓰기 가능한 건 DB 볼륨뿐입니다. api와
   dashboard는 내부 네트워크라 직접 접근이 안 되고, 격리망 운영이 전제라
   피해 반경이 컨테이너 안으로 제한됩니다.
